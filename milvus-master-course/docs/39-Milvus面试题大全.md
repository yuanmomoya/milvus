# 39 Milvus 面试题大全

## 学习目标

本章汇总 Milvus 和向量数据库相关的高频面试题，按主题分类，每题附带参考答案要点。

---

## 一、向量数据库基础

**1. 什么是向量数据库？和传统数据库有什么区别？**

向量数据库专门存储和检索高维向量，核心操作是相似度搜索（找最近邻）。传统数据库基于精确匹配（WHERE id=1），向量数据库基于近似匹配（找语义最相似的 TopK）。

**2. ANN 和 KNN 的区别？**

KNN（精确最近邻）计算查询与所有向量的距离，O(N) 复杂度，结果精确。ANN（近似最近邻）通过索引结构跳过大部分计算，用可控的召回损失换取数量级的速度提升。

**3. COSINE、IP、L2 三种度量的区别和适用场景？**

- COSINE：比较方向，归一化后与 IP 等价，文本检索常用
- IP（内积）：未归一化时受向量长度影响，推荐系统常用
- L2（欧氏距离）：比较绝对距离，图像特征常用

**4. Embedding 是什么？为什么不同模型的 Embedding 不能混用？**

Embedding 是模型将原始数据映射到向量空间的结果。不同模型的向量空间不同（坐标系不同），混用等于在不同地图上比较坐标，结果无意义。

## 二、Milvus 架构

**5. Milvus 的核心组件有哪些？各自职责？**

- Proxy：接入请求、路由
- RootCoord：DDL、时间戳
- DataCoord/DataNode：写入、Segment 管理
- QueryCoord/QueryNode：搜索执行
- IndexCoord/IndexNode：索引构建
- etcd：元数据
- MinIO/S3：数据文件
- Pulsar/Kafka：消息队列

**6. 写入成功后数据一定能被搜索到吗？**

能被搜索到（growing segment 暴力扫描），但可能不是索引搜索。索引搜索需要等 Segment seal + IndexNode 构建索引完成。

**7. Segment 的生命周期是怎样的？**

Growing → Sealed（达到阈值或 flush）→ Indexed（索引构建完成）→ Loaded（QueryNode 加载）→ Compacted（合并/清理）

**8. Standalone 和 Cluster 的核心区别？**

Standalone 所有角色在一个进程，用 RocksMQ。Cluster 各角色独立部署，用 Pulsar/Kafka，支持水平扩展和高可用。

## 三、索引

**9. HNSW 的核心思想？M 和 ef 参数的作用？**

HNSW 构建多层图，高层稀疏快速导航，底层密集精细搜索。M 控制每个节点的邻居数（影响内存和图密度），ef 控制搜索时的候选集大小（影响召回和延迟）。

**10. IVF 的 nlist 和 nprobe 如何选择？**

nlist ≈ 4×sqrt(N)。nprobe 通过实验确定：从小到大增加，找到召回率达标且延迟可接受的最小值。nprobe/nlist ≈ 5-10% 是常见起点。

**11. PQ 为什么能压缩 32 倍？代价是什么？**

PQ 将 768 维向量切成 96 段，每段用 1 字节编码（256 个聚类中心），3072B → 96B = 32×。代价是量化误差导致召回率下降。

**12. 什么时候用 FLAT、HNSW、IVF、DISKANN？**

- FLAT：< 10 万，需要精确结果
- HNSW：10 万-2000 万，内存充足，追求低延迟
- IVF：内存受限，或数据量大
- DISKANN：> 5000 万，内存极度受限

## 四、数据建模

**13. Collection 的主键应该怎么设计？**

用业务相关的确定性 ID（如内容 hash），支持 upsert 幂等。避免 auto_id（不支持 upsert）和随机 UUID（无法去重）。

**14. 什么时候用 Partition Key？什么时候用普通 filter？**

Partition Key：高频过滤维度（如 tenant_id），过滤比例高（> 80%），需要物理隔离。普通 filter：过滤条件多变，过滤比例低。

**15. enable_dynamic_field 的优缺点？**

优点：灵活，无需预定义所有字段。缺点：无法建标量索引，过滤性能差，无类型校验。建议：核心过滤字段显式定义，辅助元数据用动态字段。

## 五、性能优化

**16. 搜索延迟高怎么排查？**

按优先级：有没有索引 → ef/nprobe 是否过大 → output_fields 是否过多 → Segment 是否碎片化 → 是否有高过滤比例 → 资源是否饱和。

**17. 为什么不建议手动 flush？**

手动 flush 强制 seal 当前 growing segment，产生小 Segment。大量小 Segment 增加搜索归并开销和 Compaction 压力。

**18. mmap 的作用和代价？**

mmap 用磁盘映射代替全量内存加载，降低内存占用 50%+。代价是搜索延迟增加 2-5×（可能触发磁盘 IO）。适合内存不足但有 SSD 的场景。

**19. 批量写入的最佳实践？**

batch_size=1000-2000，2-4 线程并发，不手动 flush，带重试机制。单批数据量控制在 2-8 MB。

## 六、RAG

**20. RAG 的核心链路？**

文档解析 → 切块 → Embedding → 入库 → Query Rewrite → 向量召回 → Rerank → Prompt 构建 → LLM 生成 → 引用返回。

**21. Chunk Size 如何影响 RAG 质量？**

太小：语义碎片化，信息不完整。太大：主题混合，Embedding 不精确，噪声大。中文通用起点：400-600 字，50-100 overlap。

**22. 为什么需要 Rerank？**

向量检索是 bi-encoder（独立编码），精度有限。Rerank 用 cross-encoder（联合编码），能捕捉 query-doc 的细粒度交互，显著提升排序质量。

**23. 如何降低 RAG 的幻觉？**

Prompt 约束（只基于资料回答）+ 拒答机制（资料不足时说不知道）+ Rerank 过滤弱相关内容 + 引用标注（可追溯）+ 低 temperature。

## 七、生产运维

**24. Milvus 高可用怎么做？**

QueryNode 多副本（2+）+ etcd 三节点集群 + 可靠对象存储（S3/分布式 MinIO）+ 每日备份 + 监控告警。

**25. 模型升级时如何处理已有数据？**

新建 Collection → 双写新数据 → 离线回灌旧数据（用新模型重编码）→ 灰度切换流量 → 验证后下线旧 Collection。不能原地替换。

**26. 如何监控 Milvus？**

Prometheus 采集 `:9091/metrics` → Grafana 可视化。核心指标：搜索 P95 延迟、错误率、Segment 数量、QueryNode 内存、etcd 空间。

**27. 容量规划怎么做？**

内存 ≈ 向量数 × 维度 × 4B × 索引系数 × 副本数。HNSW 索引系数 ≈ 1.5，IVF ≈ 1.0，PQ ≈ 0.03-0.06。预留 50% 余量。

## 八、多模态与高级场景

**28. CLIP 为什么能实现文搜图？**

CLIP 通过对比学习将图片和文本映射到同一向量空间。匹配的图文对距离近，不匹配的距离远。搜索时文本向量可以直接与图片向量比较。

**29. 混合检索（Hybrid Search）的价值？**

语义检索擅长理解意图但可能漏掉精确术语，关键词检索擅长精确匹配但不理解语义。两者融合（RRF）覆盖更多查询类型。

**30. 亿级向量怎么存？**

冷热分层：热数据 HNSW（内存），温数据 IVF_SQ8（mmap），冷数据 DISKANN（磁盘）。按时间或业务分片，多 Collection 管理。

## 回答技巧

1. **先说结论，再解释原因**：面试官想快速判断你是否理解核心概念
2. **用数字说话**：延迟、内存、压缩比等具体数字比模糊描述更有说服力
3. **提到取舍**：每个技术选择都有代价，展示你理解 trade-off
4. **结合实践**：如果有实际经验，提到具体的数据规模和优化效果

## 小结

向量数据库面试的核心考察点：基础概念（向量、ANN、度量）、架构理解（组件职责、数据流）、索引原理（HNSW/IVF/PQ 的思想和参数）、工程实践（性能优化、生产运维）、应用场景（RAG、多模态）。理解原理比记忆 API 更重要。
